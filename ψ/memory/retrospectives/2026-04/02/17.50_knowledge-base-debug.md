## AI Diary

This session had a clear rhythm — build, then fix, then build again. The morning started with ambition: a full Dashboard, routing, persistence. Each piece clicked into place with satisfying precision. useRouter.js is a tiny hook, barely 20 lines, but it gives the whole app a sense of navigation that wasn't there before. useFlows.js similarly — simple localStorage CRUD, but now flows survive page refreshes. That felt like a milestone.

The drag-to-connect fix was the most satisfying debugging moment. The problem was elegant in its stupidity: the add-button (+) was positioned exactly on top of the output port, swallowing all click events before they could start a drag. Moving the button to `bottom: 8px` and the port to `top: 30%` gave each element its own territory. Then the deeper issue — even when the port was accessible, `handleMouseUp` was checking if the cursor was within a few pixels of a node's top-left corner. That's not how humans drop things. Added `getNodeAtPoint()` with full bounding box detection and suddenly connecting nodes just worked.

The knowledge base issue is the hanging thread of this session. The architecture is right — `exportConfig` sends `{name, content}`, the flow engine reads `flow.knowledge`, sets `ctx.knowledgeContext`, and the AI system prompt includes it. But the user reported the AI was "searching on its own" rather than using the file. The most likely culprit: the user hadn't redeployed since the `exportConfig` fix was applied. The old deployed flow had empty knowledge objects. Added verbose logging so the next test will immediately reveal whether knowledge content is arriving at the server or not.

## Honest Feedback

**Friction 1: The exportConfig bug lived too long.** The bug where exportConfig sent only file names (not content) was introduced during the initial refactor and wasn't caught until the user specifically tested Knowledge Base responses. The code path is: UI → exportConfig → POST /api/flow → flowEngine.executeNode(knowledge) → ctx.knowledgeContext. Testing this end-to-end should have been part of initial implementation. In the future, when adding a new data flow, trace it all the way to the consumer immediately.

**Friction 2: Server restart state.** Every time server code changes, `currentFlow` is lost because it's in-memory. The user has to redeploy from the UI to restore the flow. This is annoying during debugging — the user fixes something, restarts the server, then has to find the browser tab, click Deploy, wait. Should add server-side flow persistence (even just writing to a `flow.json` file) so restarts are seamless.

**Friction 3: No visual feedback for knowledge context.** When the user sets up a Knowledge → AI Response flow, there's no way to verify that knowledge content is actually reaching the AI without reading server logs. A "test flow" button in the UI that shows the resolved system prompt would catch this class of bug instantly. The verbose logging added today is a server-side bandaid, not a proper solution.

## Lessons Learned

1. Always trace data flows end-to-end when building pipelines — exportConfig bug would have been caught immediately if the JSON was inspected at the server on first deploy.
2. In-memory server state is a poor fit for development iteration — persisting `currentFlow` to disk would save a deployment step after every restart.
3. Visual debugging tools in the UI beat console.log — a "preview system prompt" button in the AI node config would make knowledge debugging self-serve.
